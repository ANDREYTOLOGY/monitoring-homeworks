### Задание 1.  
*Вас пригласили настроить мониторинг на проект. На онбординге вам рассказали, что проект представляет из себя платформу для вычислений с выдачей текстовых отчетов, которые сохраняются на диск. Взаимодействие с платформой осуществляется по протоколу http. Также вам отметили, что вычисления загружают ЦПУ. Какой минимальный набор метрик вы выведите в мониторинг и почему?*  

Минимальный набор метрик, которые я включил бы в мониторинг:  

`CPU usage` - потому что вычисления грузят проц  
`RAM usage` - чтобы не упереться в память  
`Disk space` - отчеты пишутся на диск, нужно контроллировать место    
`Disk IO` - чтобы понимать, не тормозит ли запись  
`HTTP метрики (RPS, latency, коды ответов)` - чтобы видеть работу сервиса  


### Задание 2.  
*Менеджер продукта посмотрев на ваши метрики сказал, что ему непонятно что такое RAM/inodes/CPUla. Также он сказал, что хочет понимать, насколько мы выполняем свои обязанности перед клиентами и какое качество обслуживания. Что вы можете ему предложить?*  

Я бы убрал тех. метрики из его дашборда и показал бизнес-понятные вещи:  
`доступность сервиса (%)` — работает или нет  
`доля успешных запросов` — сколько запросов прошло нормально  
`время ответа` — быстро или тормозит  
`количество ошибок` — сколько пользователей столкнулись с проблемами
То есть оформить это как SLA/SLO, чтобы он видел не RAM и CPU, а “нормально ли мы обслуживаем клиентов”.

### Задание 3.  
*Вашей DevOps команде в этом году не выделили финансирование на построение системы сбора логов. Разработчики в свою очередь хотят видеть все ошибки, которые выдают их приложения. Какое решение вы можете предпринять в этой ситуации, чтобы разработчики получали ошибки приложения?*  

Я бы сделал максимально простое решение без затрат:  
настроил логирование ошибок в файл (например, отдельный error.log)  
дал разработчикам доступ к этим логам (через ssh или общую папку)  
для удобства можно сделать tail -f или простой скрипт для просмотра  
плюс настроить отправку ошибок в Telegram или на почту  

То есть без сложных систем — просто централизовать ошибки и дать к ним быстрый доступ.  

### Задание 4.  
*Вы, как опытный SRE, сделали мониторинг, куда вывели отображения выполнения SLA=99% по http кодам ответов. Вычисляете этот параметр по следующей формуле: summ_2xx_requests/summ_all_requests. Данный параметр не поднимается выше 70%, но при этом в вашей системе нет кодов ответа 5xx и 4xx. Где у вас ошибка?*  

Ошибка в том, что в summ_all_requests попадают не только 2xx.
Скорее всего туда входят ещё 3xx (редиректы) или вообще лишние/не те метрики. Поэтому получается ~70%, хотя ошибок нет.
Нужно либо учитывать 3xx как успешныe, либо правильно считать общее число запросов (только реальные HTTP ответы)  

### Задание 5.  
*Опишите основные плюсы и минусы pull и push систем мониторинга.*  
```
Pull (система сама ходит за метриками):  

+ проще контролировать (всё в одном месте)  
+ легче отлаживать  
+ обычно стабильнее  

− не работает, если до сервиса нельзя достучаться  
− плохо подходит для динамичных/временных сервисов  

Push (сервисы сами отправляют метрики):  

+ удобно для временных задач и закрытых сетей  
 не нужно открывать доступ извне  

− сложнее контролировать, кто и что отправляет  
− можно потерять данные, если отправка сломалась  
− больше возни с настройкой  
```
